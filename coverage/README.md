# Coverage

The README and this project's prose have always claimed the test suite is
thorough -- 112 Catch2 tests, a three-way (`OrderBook` / `LadderBook` /
independent reference model) full-day cross-validation, and continuous
fuzzing. None of that was ever backed by a measured number: no gcov/lcov
integration, no coverage badge, nothing that turns "trust me, it's well
tested" into an artifact you can point at. This directory is that number.

## What's here

- [`make_badge.py`](make_badge.py) -- turns an lcov tracefile into a
  line/branch coverage percentage, a markdown table (for the CI job
  summary), and a shields.io badge JSON.
- `coverage.json` -- the committed badge source (shields.io "endpoint"
  schema). CI overwrites this after every push to `main`; it's what the
  coverage badge in the top-level README points at.

`lcov.info` / `lcov.filtered.info` are **not** committed -- they're
multi-megabyte generated tracefiles, regenerated by every CI run (and by
you locally, see below). Only the small `coverage.json` summary is tracked,
the same treatment `bench/ci_history.csv` gets for benchmark history versus
its own raw per-run data.

## How it's wired into CI

`.github/workflows/ci.yml` has a `coverage` job, separate from `build-test`
and `sanitize`, because it needs its own build: `--coverage` instrumentation
(`-fprofile-arcs -ftest-coverage`, GCC's flag on the `ubuntu-latest`
default-`g++` toolchain) changes codegen and would be wasted work bolted
onto any of the other jobs' builds.

1. **Configure + build** a dedicated `build-coverage` tree with
   `-DCMAKE_CXX_FLAGS="--coverage -O0" -DCMAKE_EXE_LINKER_FLAGS="--coverage"`
   (all targets, not just `unit_tests` -- `replay_selftest`'s `ctest` case
   execs the real `replay` binary, same reason `build-debug` in the
   `build-test` job builds everything).
2. **Run the full suite** (`ctest --test-dir build-coverage`) -- every
   `.gcda` file it produces is this run's coverage data.
3. **Capture + filter with lcov**: `lcov --capture` over the build tree,
   then `lcov --remove` to drop system headers, the FetchContent-vendored
   Catch2 tree (`_deps/`), and the test files themselves (`tests/*` --
   coverage of test code isn't a meaningful signal; the point is how much
   of `include/` + `src/` the tests exercise).
4. **`make_badge.py`** reads the filtered tracefile, prints a markdown
   table to `$GITHUB_STEP_SUMMARY` (visible directly on the Actions run
   page), and writes `coverage/coverage.json`.
5. **Commit the updated `coverage.json` back to `main`** -- push events
   only, never PRs (a PR from an unmerged branch shouldn't get to overwrite
   the badge everyone else sees) -- using the same
   configure-git-identity / `git diff --staged --quiet` guard /
   pull-rebase-push retry loop `ci.yml`'s "Commit updated ci_history.csv"
   step already uses for benchmark history. See that step (and
   `bench/CI_INTEGRATION.md`) for why the retry loop exists: both matrix
   legs of `build-test` can race to push, and this job can race them too.

Nothing here fails the build on a coverage drop -- there is no threshold
gate, deliberately. A hard coverage-percentage gate invites exactly the
kind of test-padding (asserting on lines instead of behavior just to move
a number) this fix is trying to avoid; the goal is a real, visible number
replacing an unverifiable claim, not a new thing to game. If a future
change wants a gate, `make_badge.py`'s `line_pct`/`branch_pct` are already
computed and easy to threshold against `sys.exit(1)`.

## Reading the badge

[![Coverage](https://img.shields.io/endpoint?url=https://raw.githubusercontent.com/Manav0559/itch-lob-engine/main/coverage/coverage.json)](https://github.com/Manav0559/itch-lob-engine/actions/workflows/ci.yml)

The badge (also in the top-level README) reads `coverage/coverage.json` off
`main` via shields.io's
[endpoint badge](https://shields.io/badges/endpoint-badge) support --
no Codecov/Coveralls account, upload token, or third-party service, in
keeping with this project's zero-external-dependency CI (same reasoning as
`dashboard/`'s hand-rolled inline-SVG charts instead of a JS charting
library). It shows both line and branch coverage, since the two disagree
enough on this codebase (branch coverage is meaningfully lower --
error-path/gap-handling branches in the networking and pipeline code are
the biggest gap) that a single number would hide which one you're looking
at.

## Running it locally

Needs `lcov` (`brew install lcov` on macOS, `apt-get install lcov` on
Debian/Ubuntu) and a GCC-or-Clang toolchain that supports `--coverage`
(both do; Clang's driver accepts `--coverage` as an alias and emits the
same `.gcno`/`.gcda` format GCC's gcov and lcov consume).

```bash
cmake -B build-coverage -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_CXX_FLAGS="--coverage -O0" \
  -DCMAKE_EXE_LINKER_FLAGS="--coverage"
cmake --build build-coverage --parallel
ctest --test-dir build-coverage --output-on-failure

lcov --capture --directory build-coverage --output-file coverage/lcov.info \
  --rc branch_coverage=1
lcov --remove coverage/lcov.info '/usr/*' '*/_deps/*' '*/tests/*' \
  --output-file coverage/lcov.filtered.info --rc branch_coverage=1

lcov --list coverage/lcov.filtered.info --rc branch_coverage=1   # per-file table
python3 coverage/make_badge.py coverage/lcov.filtered.info       # totals + badge
```

`coverage/lcov.info` and `coverage/lcov.filtered.info` are gitignored --
they're regenerated, multi-megabyte, and machine-specific (absolute paths
baked in by `geninfo`); don't commit them.

On a Clang-only machine without GCC (e.g. stock macOS), `lcov`'s
`geninfo` needs to detect the LLVM `gcov` shim (`xcrun llvm-cov gcov` on
macOS with Xcode Command Line Tools installed) and may emit extra
`inconsistent`/`unsupported`/`format` warnings that real GCC-generated
`.gcda` data doesn't trigger -- harmless (`lcov --ignore-errors
inconsistent,unsupported,format,...` silences them), and not something
`ci.yml` needs to work around since its `coverage` job uses the
`ubuntu-latest` runner's real `g++`/`gcov`.
